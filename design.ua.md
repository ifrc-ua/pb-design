# Огляд дизайн-системи Бюджет участі-ІФ

[🇬🇧 Read in English](design.md)

## TL;DR (для AI-агентів і нетерплячих читачів)

- **Бренд-кольори**: фіолетовий `#654EA3` + жовтий `#FFEC08` на майже-білому `#FDFDFD`, текст `#1A1A1A`.
- **Шрифти**: `Phenomena` для дисплею/заголовків/великих чисел; `Proxima Nova` для решти. Замінники (коли комерційні шрифти недоступні): `Inter Tight` + `Inter` через Google Fonts CDN.
- **Числа**: завжди `tabular-nums`; роздільник тисяч — нерозривний пробіл (`14 832`, ніколи `14,832`); десятковий роздільник — кома (`3,7%`).
- **Правило жовтого**: жовтий — акцент, ніколи не великий фон. У графіках жовті сегменти завжди з обведенням `#1A1A1A` 1.5px.
- **Layout**: mobile-first (спершу тестуємо на 375px); радіуси `8/12px` для кнопок/карток; `96px` між секціями; `tabular-nums` скрізь де є цифри.
- **Сфера охоплення**: 10 років історичних даних БУ за період 2016–2026 (у 2022 році БУ не проводився). Не активні цикли голосування, без dark mode, без адмін-UI, без форм подачі.
- **YAML > проза**: при конфлікті YAML-frontmatter і прози — джерело правди YAML.

Повний документ нижче пояснює *чому* за кожним правилом. Прочитайте раз, потім повертайтеся за довідкою.

Цей документ визначає візуальні стандарти та правила побудови інтерфейсів для екосистеми Бюджету участі Івано-Франківська. Система оптимізована для роботи з AI-агентами, створення аналітичної інфографіки/візуалізацій, муніципальних звітів тощо.

## 1. Візуальна тема і атмосфера

Бюджет участі (громадський бюджет) — це муніципальний інструмент, який живе на перетині двох енергій: **громадянської довіри** (спокій, прозорість, порядок) та **активної участі** (рух, голос, вибір). Дизайн-система побудована навколо цього дуалізму.
Використано два шрифти:

**Phenomena** — умовний, широкий, майже архітектурний гротеск. Виступає в ролі голосу-героя: в дисплейних заголовках, рубриках, цифрах. Його геометричність і прозорий контур асоціюються з громадським простором, офіційністю без холодності.
**Proxima Nova** — універсальна робоча гарнітура. Відповідає за щільні шари інтерфейсу, аналітичні підписи, довгі описи проєктів і таблиці даних.

Палітра тримається на двох кольорах з логотипу:
**фіолетовий `{colors.primary-500}`** (`#654EA3` — серйозність, влада, креативність)
**жовтий `{colors.secondary-500}`** (`#FFEC08` — енергія, голос, видимість). Фіолетовий як чорнило документа, жовтий як маркер для найважливішого. Фон майже білий (`{colors.canvas}`), текст глибокий графітовий (`{colors.ink}`), що є парою для максимальної читабельності і фокусу на даних. Ненасичена дисципліна кольору означає, що у ядрі тримаємо тільки два бренд-кольори + сірі нейтрали; розширена data-viz-палітра (коли треба передати 5+ категорій на одному графіку) відображена в окремому `design-data.md`.

Естетика стриманої інфографіки передбачає багато білого простору, чіткі сітки, великі числа як героїв композиції, акуратні іконки з товщиною 1.5px, м'які (8–12px) заокруглення на картках, тіні ледве помітні. Жодних неонових градієнтів, жодних декоративних анімацій, усе слугує читанню даних і зрозумілості.

**Ключові характеристики:**

- Дві гарнітури: Phenomena (дисплей + заголовки) + Proxima Nova (UI, текст, дані)
- Двоколірний логотипний бренд: фіолетовий `{colors.primary-500}` + жовтий `{colors.secondary-500}`
- Data-viz у ядрі — Primary shades + жовтий. Повна категорійна палітра (11 канонічних категорій) живе в [design-data.md](./design-data.md).
- Історичний, аналітичний фокус — система створена для візуалізації **10 років БУ за період 2016–2026** (у 2022 році БУ не проводився), а не для активного циклу голосування.
- Жовтий `{colors.secondary-500}` — завжди маркер важливого (переможець `{components.badge-winner}`, акцент, CTA), ніколи фон великих поверхонь
- Акуратні тіні, сітка `{spacing.xs}` (8px), широкі міжколонки — муніципальна довіра = «не тісно»
- Mobile-first: сайт і соцмережі споживають з телефону

## 3. Типографіка

### Сімейства шрифтів

- **Display/Headings**: `Phenomena` (комерційна ліцензія, файли не включені), fallback: `'Inter Tight', system-ui, sans-serif`
- **Body/UI**: `Proxima Nova` (комерційна ліцензія, файли не включені), fallback: `'Inter', -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif`
- **Numerics/Data**: `Proxima Nova` з `font-variant-numeric: tabular-nums` — обов'язково для таблиць, дашбордів, графіків

### Замінники шрифтів (коли комерційних немає)

Зовнішні AI-інструменти (Lovable, v0, Stitch, Canva) і презентації третіх сторін не містять Phenomena чи Proxima Nova. Використовуйте ці замінники і обов'язково застосовуйте перелічені корекції, інакше типографічний ритм розсиплеться.

| Оригінал | Замінник | Корекція |
|---|---|---|
| Phenomena (дисплей, заголовки) | `Inter Tight` 900 | Inter Tight має ширші чаші літер; зменште letter-spacing ще на `-0.5px` на розмірах 64px+. |
| Proxima Nova (UI, body) | `Inter` (regular/semibold/bold) | Висота рядка Inter на 2–3% більша; підтягніть `body-md` line-height з 1.55 → 1.50, `body-lg` з 1.55 → 1.45. |
| Proxima Nova `tabular-nums` | `Inter` з `font-variant-numeric: tabular-nums` | Обов'язково пропорційні цифри Inter за замовчуванням розваляють вирівнювання таблиць. |

> Geist Sans/Geist Mono — прийнятний другий резерв, якщо Inter теж недоступний.
> Ніколи не замінюйте Phenomena серифом, slab-серифом чи дисплейним скриптом, адже система втратить свою архітектурну стриманість.

### Локальні шрифти (в репо, gitignored)

Коли працюєте з цим репозиторієм **локально**, реальні брендові файли шрифтів доступні на диску:

- `fonts/Phenomena/` — 7 насиченостей (Thin → Black), `.otf`
- `fonts/Proxima-Nova/` — 13 файлів (Thin → Black + italics), `.otf`

**Використовуйте їх для всього, що рендериться локально** (HTML/CSS через `@font-face` з відносним шляхом, matplotlib через `font_manager.fontManager.addfont()`, Pillow/Cairo через `ImageFont.truetype("fonts/Phenomena/Phenomena-Black.otf", size)` тощо) — справжні шрифти мають пріоритет над замінниками Inter Tight / Inter вище.

### Responsive Type Scale

Значення у YAML-frontmatter — **desktop defaults**. AI-агенти повинні масштабувати їх на менших breakpoints, інакше заголовки ламають viewport на мобільному (96px hero на 375px екрані переноситься посеред літери).

| Токен | Mobile (≤599px) | Tablet (600–1023px) | Desktop (1024–1439px) | Wide (1440px+) |
|---|---|---|---|---|
| `display-hero` | 56px | 72px | 96px | 96px |
| `data-stat` | 48px | 56px | 64px | 72px |
| `title-h1` | 36px | 48px | 56px | 64px |
| `title-h2` | 28px | 32px | 36px | 40px |
| `title-h3` | 22px | 24px | 26px | 28px |
| `title-feature` | 18px | 20px | 20px | 22px |
| `body-lg` | 17px | 18px | 19px | 20px |
| `body-md` | 16px | 16px | 16px | 16px |
| `body-sm` | 14px | 14px | 14px | 14px |
| `caption` | 12px | 12px | 13px | 13px |
| `label-bold` | 11px | 11px | 12px | 12px |
| `data-label` | 12px | 12px | 13px | 13px |

**Правила:**

- Основний текст (`body-md`/`body-sm`) не масштабується: читабельність однакова на всіх пристроях.
- Letter-spacing масштабується разом із розміром: на мобільному (≤599px) зменшуйте display/title `letter-spacing` ближче до 0 (наприклад, `display-hero` -2px → -1.2px на мобільному). Щільний tracking погано виглядає на дрібному тексті.
- Line-height залишається таким як визначено у YAML на всіх breakpoints.
- Для довільних ширин між breakpoints використовуйте `clamp()` замість фіксованих кроків: `font-size: clamp(56px, 8vw, 96px)` для hero.

### Принципи

- **Phenomena — лише для заголовків і великих чисел.** Ніколи не використовуйте Phenomena для довгих абзаців, її широкий контур руйнує читабельність.
- **Proxima Nova — для всього, що довше ніж 5 слів.** Весь UI, описи, списки, таблиці, дані.
- **Tabular numerals обов'язкові** для всіх чисел у таблицях, дашбордах і графіках, щоб цифри вирівнювалися вертикально.
- **Uppercase з letter-spacing +1px** — тільки для Overline/Label. Не робіть uppercase у заголовках H1–H3.
- **Контраст ваг — важливий:** у великих числах використовуйте 900 Black для максимального ваги; у body — 400 Regular.
- **Italic використовувати обережно:** тільки для цитат, виносок «прим.», назв сторонніх джерел.

### Форматування чисел і валюти (українська локаль)

Усі числа форматуються за українським стандартом. Це **обов'язково** — англомовні агенти (Stitch, Lovable, v0) за замовчуванням ставлять англійські коми, і без цього розділу ваші «14 832 голоси» перетворяться на «14,832».

- **Розряди тисяч**: нерозривний пробіл (` `). `14 832`, `127 456`, `1 412`. НЕ `14,832`, НЕ `14.832`.
- **Десятковий роздільник**: кома. `3,7%`, `5,2 млн`. НЕ `3.7%`.
- **Валюта**: `₴` перед числом з нерозривним пробілом — `₴5,2 млн`. Для точних сум — постфіксом: `62 437 ₴`.
- **Скорочення**: `тис.`, `млн`, `млрд` через нерозривний пробіл. `127 тис.`, `5,2 млн`.
- **Відсотки**: тісно, без пробілу. `47,3%`.
- **Діапазони років**: en-dash без пробілів. `2016–2026`, НЕ `2016-2026`, НЕ `2016 - 2026`.
- **Роки**: повною цифрою. `2019`, НЕ `'19`.
- `tabular-nums` — обов'язково на всіх числах у UI, таблицях і графіках (див. Hierarchy вище).

## 5. Принципи макета

### Система відступів (8px база)

`4px, 8px, 12px, 16px, 20px, 24px, 32px, 40px, 48px, 64px, 80px, 96px`

> YAML-frontmatter (`spacing.xxs … spacing.section`) називає лише вісім найвживаніших семантичних стопів (4 / 8 / 12 / 16 / 24 / 32 / 48 / 96px). Проміжні значення **20, 40, 64, 80px** так само належать 8px-сітці, але посилаються на них у CSS сирим піксельним значенням — вони існують для разового композиційного підстроювання (наприклад, 80px відступ між hero-смугою і наступною карткою). Не вигадуйте кроків поза сіткою.

### Сітка та контейнер

- Максимальна ширина контейнера: **1280px**
- Бічні відступи: 24px (мобільний), 40px (планшет), 64px (десктоп)
- Колонки: 4 (мобільний), 8 (планшет), 12 (десктоп)
- Проміжки: 16px (мобільний), 24px (планшет+)

### Філософія білого простору

- **Муніципальна довіра = не тісно.** Секції розділяються `{spacing.section}` (96px) вертикального простору на десктопі (`{spacing.xxl}` для мобільних). Дозволяйте сторінці повертатися до білого полотна `{colors.canvas}` між кожними двома секціями, щоб кожен блок читався як відокремлений.
- **Картки дихають:** відступ усередині картки `{spacing.lg}` (24px), між картками у сітці — `{spacing.lg}`.
- **Дані мають повітря:** графіки й таблиці мають відступ 24–32px від країв контейнера.

### Патерн поверхонь секцій (кольорові блоки)

Лонгрідні сторінки («10 років БУ», глибокі огляди проєктів) потребують вертикального ритму. Система чергує поверхні, а не декорує, щоб розбивати сторінку на секції.

**Дозволені поверхні:**

- `{colors.canvas}` (#FDFDFD) — за замовчуванням, сторінка завжди повертається сюди між двома кольоровими блоками.
- `{colors.primary-50}` (#F6F4FB) — спокійний фіолетовий блок для статистичних секцій, наративних підсумків, «загального плану».
- `{colors.primary-100}` (#EEEAF7) — трохи насиченіший фіолетовий для блоків-акцентів (наприклад «Топ-переможець десятиліття»).
- `{colors.secondary-100}` (#FFF9B3) — жовтий блок, використовується **щонайбільше раз на сторінку**, лише для одного найважливішого моменту (hero-число, фінальна CTA-смуга).

**Правила:**

- **Один кольоровий блок на видимий екран.** Якщо два кольорові блоки одночасно потрапляють у viewport висотою 1080px, між ними обов'язково встав canvas-секцію.
- **Canvas завжди повертається.** Ніколи не ставте дві кольорові секції підряд.
- **Кольорові блоки на мобільному — full bleed** (без бічних відступів, без заокруглень). На десктопі як `rounded-xl` контейнери з `{spacing.section}` (96px) внутрішнього padding (компонент `section-hero`).
- **Тип не змінюється.** Зміна поверхні не дозволяє нові шрифти чи нові кольори токенів — змінюється лише фон.
- **Жовті блоки вимагають стриманості.** На жовтій секції `button-primary` усередині отримує темну рамку (`1px solid {colors.ink}`) для контрасту.

> Цей патерн є інструментом темпу системи. 6-секційна сторінка має читатися приблизно як: canvas → primary-50 → canvas → primary-100 → canvas → secondary-100 (фінальна CTA). Ніколи: primary-50 → primary-100 → primary-50 поспіль.

### Шкала заокруглень

- `4px` — маленькі елементи (chip-label усередині, progress ticks)
- `8px` — кнопки, inputs, маленькі картки
- `12px` — картки проєкту, стандартні блоки
- `16px` — великі stat-картки, модалки
- `24px` — великі hero-секції, featured-cards
- `999px` (pill) — чіпи, бейджі, тонкі кнопки-tags

### Геометрія фото та зображень

Реальні фото проєктів БУ дуже різні за якістю — система маскує це уніфікованою геометричною рамкою.

- **Пропорції**: 3:2 у картках, 16:9 у hero, 1:1 лише для соцмережних квадратних плиток. Ніколи не довільні.
- **Заокруглення**: `{rounded.sm}` (8px) усередині `card-project` (зовнішнє 12px); `{rounded.md}` (12px) для самостійних hero-зображень; `{rounded.lg}` (16px) для full-bleed hero-фото в `section-hero`.
- **Без тіней на фото.** Фото лежить пласко на поверхні картки.
- **Без поворотів і параллаксу.** Фото — статичні прямокутники. Це муніципальна репортажність, а не редакційний дизайн.
- **Без аватарок.** На авторів посилаємось іменем + район (Proxima Nova 14px, neutral-500), ніколи фото, щоб не порушувати приватність мешканців.
- **Lazy-loading обов'язковий** для будь-яких зображень нижче першого viewport — `loading="lazy"` на кожному `<img>` поза hero.
- **Object-fit: cover** з `object-position: center` за замовчуванням. Ручне кадрування дозволено лише коли в кадрі є обличчя або критична деталь, яка має лишитись.
- **Заглушка коли фото немає**: фон `{colors.primary-50}`, по центру 32px-іконка `camera-off` (Lucide) кольору `{colors.neutral-400}`. Ніколи не вставляйте стокові фото як заглушку.
- **Alt-текст обов'язковий** — описуйте об'єкт проєкту («двір з новим майданчиком на Галицькій»), а не композицію кадру.

## 7. Що робити і що не робити

### Робіть

- Використовуйте Phenomena лише для заголовків і великих чисел.
- Використовуйте Proxima Nova для всього UI, тексту, даних і підписів.
- Застосовуйте `tabular-nums` на всіх цифрах у таблицях і дашбордах.
- Жовтий `#FFEC08` — це **акцент**, не фон. Використовуйте для позначення переможця, хайлайтів, винятково важливих CTA.
- На графіках з жовтим завжди додавайте темний контур `#1A1A1A` 1–1.5px.
- У ядрі data-viz — Primary shades + жовтий як акцент. Категорійна палітра для багатосегментних графіків (donut, stacked bar, streamgraph) живе в [design-data.md](./design-data.md) §4.
- Для heatmap / choropleth використовуйте градацію одного кольору (sequential) — найприродніше Primary-50 → Primary-900.
- Давайте секціям 80–96px вертикального простору.
- Округлення кнопок — 8px, карток — 12px, pill для чипів/бейджів.
- Перевіряйте контраст тексту/фону на WCAG AA (мін. 4.5:1 для body, 3:1 для великого тексту).

### Не робіть

- Не використовуйте Phenomena для body-тексту — читабельність провалиться.
- Не використовуйте фіолетовий на жовтому або білий на жовтому — обидва провалюють AA.
- Не робіть великі заливки жовтим — він стомлює око і створює ефект попередження.
- Не використовуйте неонові або градієнтні кольори — бренд стриманий.
- Не округлюйте великі цифри в офіційній статистиці — «14 832 голоси», не «~15 тис.».
- Не плутайте ролі: «живі» статусні чипи типу «Триває голосування» — **забороняються**. Сайт — це **аналітика минулого**, не поточний цикл БУ.
- Не використовуйте бокові тіні (x-offset), тільки вертикальні.
- Не робіть dark-mode варіанти — муніципальна прозорість базується на світлому інтерфейсі; темна тема **поза межами** цієї системи.
- Не робіть inline-редагування тексту шрифтом Phenomena 300 або тонше — його Thin/Light для дисплейного застосування тільки у великих розмірах (48px+).

## 8. Адаптивна поведінка

### Точки перелому

| Ім'я | Ширина | Ключові зміни |
|---|---|---|
| Mobile | 360–599px | 1 колонка, стек, hero-число 48px, padding 24px |

| Wide | 1440px+ | Максимальний контейнер 1280px, центроване |

### Зони дотику (touch targets)

Mobile-first означає реальні пальці на реальних телефонах у реальних маршрутках. Кожен інтерактивний елемент мусить виконувати ці мінімуми.

| Елемент | Мобільний (тач) | Десктоп (миша) |
|---|---|---|
| Pill-кнопки (primary, secondary, accent) | мін. висота **44px** | 40px прийнятно |
| Поля вводу (dropdown, text, select) | мін. висота **48px** | 40px прийнятно |
| Чекбокс / тогл — зона дотику | **44×44px** (візуально може 20px, зона збільшена padding'ом) | 24×24 прийнятно |
| Handle повзунка років | **24px** видимий діаметр + зона дотику **44×44px** | 16px видимо, зона 32×32 |
| Іконкова кнопка (наприклад закриття чипа, zoom мапи) | **44×44px** | 32×32px |
| Inline-посилання (всередині абзацу) | line-height ≥ 1.55 + 8px вертикального padding на тапі | н/д |
| Маркери на мапі | мін. зона тапу **32×32px** (візуальна іконка може 20px) | 20×20px |

**Відстань між сусідніми зонами**: ≥ 8px проміжку, щоб великий палець не промахувався.
**Ручки bottom-sheet з фільтрами**: на всю ширину, висота ≥ 56px.

> Тестуйте на 375px з оверлеєм пальця (коло 44px) перш ніж заявляти, що екран готовий до мобільного.

### Стратегія стиснення

- Hero display: 96px → 72px → 56px → 48px
- H1: 64px → 48px → 36px
- Section H2: 40px → 32px → 28px
- Grid колонок: 12 → 8 → 4
- Мапа: збирає фільтри в bottom-sheet на mobile
- Таблиці 10-річних даних: конвертуються у горизонтальний прокрут або стек-картки на мобільному
- Навігація: згорнута у бургер-меню на мобільному

### Пріоритет мобільного

Сайт 10 років БУ і всі інфографіки **дизайняться спершу під mobile** — бо більшість громадян споживає муніципальний контент з телефону. Карта, фільтри, статистика мають залишатися зручними на екрані 375px.

## 10. Голос і тон (редакційний стиль)

### Принципи

- **Формальний-але-теплий.** «Ви» — завжди. «Ти» — ніколи, навіть у соцмережах.
- **Активні дієслова, конкретні результати.** «Мешканці Пасічної обрали 14 проєктів» — краще ніж «було проведено голосування». «Відремонтовано 12 дворів на 5.2 млн грн» — краще ніж «виконано низку заходів з благоустрою».
- **Без канцеляризмів.** Забороняються: «вищезазначений», «в рамках», «з метою», «шляхом», «здійснення». Пишіть простіше в стилі plain language, але не примітивно.
- **Числа — героїв не ховаємо.** Якщо за 10 років брало участь 127 тисяч людей — це має бути велике число, не абзац.
- **Акцент на людях і результатах, не на процесі.** «Марія з Автовокзалу подала перший у місті проєкт велостанції» — краще ніж «Подання заявок відбувалося через електронний кабінет».

### Заголовки

- Короткі, конкретні, з числом або іменем, якщо можливо.
- ✓ «10 років, 1 412 проєктів, 8 переможців-легенд»
- ✗ «Огляд реалізації проєктів бюджету участі за десятирічний період»

### Підписи графіків

- Пояснюйте **висновок**, не опис: «Освітні проєкти виграли 5 із 6 місць у 2018-му» — краще ніж «Розподіл переможців за категоріями у 2018 році».

### Соцмережі

- Короткі факти з емоційним гачком: «У 2019 двір на Галицькій отримав більше голосів, ніж будь-який проєкт освіти за всі 10 років. Як так вийшло — дивіться в інсайтах ↓»
- Завжди пропонуйте конкретну дію: посилання на сайт з фільтром.

## 12. Рух та взаємодія

Рух у цій системі служить **моментам акценту**, тобто появі великих чисел, входу графіків, hover-станам. Базова позиція інтерфейсу — нерухомість. Цей розділ задає мінімальні правила, щоб агенти не вигадували довільних анімацій і щоб сигнатурне «намотування числа при прокруті» відтворювалося однаково.

### Базові правила

- **`prefers-reduced-motion` — обов'язково.** Усі анімації мусять мати CSS-fallback, що показує фінальний стан без руху.
- **Нерухомість — за замовчуванням.** Анімуємо лише в моментах: перша поява елемента у viewport, hover/focus, зміна стану (dropdown, таби, модалки).
- **Рух служить даним.** Не анімуємо декоративно — тільки щоб звернути увагу на число або перехід між станами.

### Тривалість та пом'якшення

| Контекст | Тривалість | Пом'якшення |
|---|---|---|
| Hover / focus (колір, тінь) | 150ms | `ease-out` |
| Підказка, випадний список, перемикання вкладок | 200–250ms | `cubic-bezier(0.22, 1, 0.36, 1)` |
| Поява картки при прокруті | 400–500ms | `cubic-bezier(0.22, 1, 0.36, 1)` |
| **Count-up для hero-чисел** | 900–1200ms | `cubic-bezier(0.25, 0.1, 0.25, 1)` |
| Вхід графіку (бари ростуть, лінії малюються) | 800–1000ms | `cubic-bezier(0.22, 1, 0.36, 1)` |
| Відкриття модального вікна | 200–250ms (fade + scale) | `ease-out` |

### Count-up великих чисел (сигнатурна анімація)

Це підпис-момент системи. Коли hero-число (Phenomena Black 48px+) вперше потрапляє у viewport — воно намотується з `0` до цільового значення.

- **Тригер**: `IntersectionObserver`, threshold 0.4 (коли 40% елемента вже видно)
- **Тривалість**: 900–1200ms (довші числа — ближче до 1200)
- **Пом'якшення**: природне сповільнення — швидкий старт, гальмування в кінці
- **Форматування проміжних значень**: також через нерозривний пробіл (`8 421` → `14 832`), не «14832» без пробілу
- **Мінімум**: не застосовувати на числах `< 100` виглядає дрібно
- **Кілька stat-cards поруч**: стагер 80–120ms між картками, не одночасно

### Анімації входу графіків

- **Bar**: бари ростуть знизу до фінальної висоти, 800ms, стагер 50ms між барами
- **Line / Area**: лінія малюється зліва направо через `stroke-dashoffset`, 1000ms. Area-заповнення — fade-in після того, як лінія намалювалася
- **Donut / Pie**: сегменти малюються від 12 години за годинниковою стрілкою, 800ms total
- **Heatmap**: клітинки fade-in зліва-вгору → право-вниз, стагер 15ms між клітинками
- **Тригер**: завжди при вході у viewport, не при перезавантаженні сторінки

### Інтерактивні стани

- **Hover на картці**: `translateY(-2px)` + shadow level 1 → level 2, 150ms
- **Hover на bar/dot графіка**: заливка +10% світліша, tooltip fade-in 150ms
- **Фокус-кільце**: `outline: 2px solid #654EA3; outline-offset: 2px` — **без анімації** (миттєво, для доступності)
- **Клік на кнопку**: `scale(0.98)` на 100ms, потім повернення — легкий тактильний відгук

### Зменшений рух

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

У цьому режимі: count-up показує **одразу фінальне число** без намотування, chart entry — фінальний стан без малювання.

### Не робіть

- Не анімуйте background colors (наприклад `white → purple` для «вжух-ефекту») — нервує.
- Ні parallax-скрол, ні bounce / elastic easing — це тон ігрового сайту, не муніципального.
- Count-up довший за 1.5s — користувач уже переключив увагу.
- Typewriter-анімація заголовків (літери «друкуються» по черзі) — проблемна для дислексії.
